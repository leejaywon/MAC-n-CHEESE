## 1. The Ralph Loop (what's actually being judged)

Source: [ghuntley.com/ralph](https://ghuntley.com/ralph/), [how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)

Core: `while :; do cat PROMPT.md | <agent> ; done` — fresh context each iteration, **one task per iteration**, progress lives in files + git, not in context.

Canonical file layout:
- `PROMPT.md` — per-iteration instructions (pick ONE task, do it, test, commit, exit)
- `specs/` — the product spec; the main steering lever. **Fix the spec, not the loop.**
- `fix_plan.md` — prioritized task list; regenerated by planning iterations
- `AGENT.md` — how to build/test/run; agent updates it with discoveries

Key techniques:
- **Backpressure:** wire validation (tests, compilers, linters, eval scores) that reject bad output. This is the #1 quality lever.
- Subagents for expensive reads (search/summarize); only 1 agent for build/validate.
- Sign-posting: when the agent repeatedly fails the same way, add explicit warnings to PROMPT.md ("search before assuming not implemented", "NO placeholder implementations").
- Commit every green iteration; `git reset --hard` is the recovery valve.
- Planning loops vs building loops as separate modes.

Known failure modes: hallucinated "not implemented" conclusions, duplicate implementations, placeholder code, context bloat, wedged builds.

### Past winners (what wins)
- **Seoul (Feb 2026):** winner *houseops* — 169k LOC, 100% autonomous, IP-camera room-cleanliness agent; set loop at 8pm, slept. 2nd: 21 user stories in 7h. ([team-attention.com](https://www.team-attention.com/en/ralphthon))
- **SF (Mar 2026):** winner *Orchid* — captured AI-code reasoning that's normally trapped on local machines. ([eomag.io](https://www.eomag.io/article/ralphthon))
- Pattern: winners had **superior harness design** (custom constraints, self-validation), a real working artifact by morning, and a clean story. Tools used: Codex (5 of 7 Seoul teams), Claude Code (2).

### Agent choice
- **Codex CLI** (`codex exec`): sponsor's tool; presenting partner. Non-interactive mode streams progress to stderr, final message to stdout; `--json` for events; must set `--sandbox workspace-write` (or danger-full-access) + `--ask-for-approval never` or the loop stalls. API-key auth for unattended runs. ([developers.openai.com/codex/noninteractive](https://developers.openai.com/codex/noninteractive))
- **Claude Code** (`claude -p` in a loop): equally viable, Ralph's original substrate.
- Both allowed historically. Using Codex has sponsor-alignment upside; prizes are OpenAI API credits either way.

## 2. Track 1 — AI Scientist

### Prior art
| System | Takeaway |
|---|---|
| [AI Scientist-v2](https://arxiv.org/abs/2504.08066) (Sakana) | Full pipeline: hypothesis → experiments (agentic tree search run by an experiment-manager agent) → figures w/ VLM feedback → manuscript. First AI paper accepted at an ICLR workshop. ~$6–15/paper. |
| [Agent Laboratory](https://github.com/SamuelSchmidgall/AgentLaboratory) (MIT license) | 3 phases: lit review → experimentation → report writing; open source, adaptable. 84% cost reduction vs. prior systems. |
| [AgentRxiv](https://agentrxiv.github.io/) | Agents share/build on each other's papers; showed iterative improvement on MATH-500. |
| [CycleResearcher](https://github.com/zhu-minjun/Researcher) | Research agent trained with a *reviewer* model as reward — generator↔reviewer loop. |
| ["Why LLMs Aren't Scientists Yet"](https://arxiv.org/abs/2601.03315) | 4 end-to-end attempts, 3 failed. Failure modes: training-data bias, implementation drift, memory degradation, **premature success declarations**, weak experimental judgment. |

### Options

**T1-A: Full pipeline from scratch (AI Scientist-v2 clone).**
The loop builds idea→experiment→paper end to end. Highest ambition, highest risk: known failure modes + 6-hour budget + GPU experiments that can silently fail. Likely ends with a broken pipeline and no PDF.

**T1-B: Fork Agent Laboratory / AI-Scientist-v2 and adapt.**
Fast start, but weak "the agent built it" story — judges at a Ralph event want to see what the *loop* produced, and adapting someone else's 10k-LOC pipeline autonomously is its own tar pit.

**T1-C: Micro-paper factory (recommended for this track).**
Constrain the science to a domain where experiments are *cheap, fast, and verifiable*, so the loop can complete multiple full research cycles in hours:
- LLM behavioral studies via API (prompt ablations, tokenizer quirks, sampling-strategy comparisons, agent-failure taxonomies)
- Small-model experiments on the VESSL GPU (nanoGPT-scale ablations, ~minutes per run)
- Every experiment logged to W&B → real figures for free
- Output: 4-page workshop-style LaTeX paper, compiled PDF as the demo artifact
- **Reviewer-in-the-loop as backpressure:** an ICML-style reviewer agent scores each draft; the loop iterates until the score clears a bar. This directly answers the event's framing ("better research loops") and mirrors CycleResearcher.
- Guards against the documented failure modes: results must be parsed from actual run logs (no "premature success"), experiments re-run for confirmation, claims must cite a specific W&B run.

### Success criteria for demo
Compiled PDF(s) with real figures from real runs + W&B dashboard showing every experiment + git history proving zero human touches.

## 3. Track 2 — Review Agent

### Prior art & assets
- [OpenReviewer](https://github.com/maxidl/openreviewer): fine-tuned 8B on 79k reviews; key finding — generic LLMs are **too positive**; critical calibration is the differentiator.
- [DeepReviewer](https://github.com/zhu-minjun/Researcher) + [DeepReview-13K dataset](https://huggingface.co/datasets/WestlakeNLP/DeepReview-13K): staged "deep thinking" review process; 64% accept/reject accuracy on ICLR 2024; 14B beats 70B via process design.
- [FLAWS benchmark](https://arxiv.org/abs/2511.21843): 713 papers with injected claim-invalidating errors; best frontier model only 39.1% error identification → **huge headroom, quantifiable**.
- Eval data with ground truth: ICLR OpenReview dumps ([berenslab/iclr-dataset](https://github.com/berenslab/iclr-dataset), ICLR-2k), metrics = decision accuracy, score MSE / Spearman vs. human review averages.
- Context: ICML 2026 itself just introduced a policy permitting LLM-assisted reviewing — topical, judges (many are actual ICML reviewers) will care.

### Options

**T2-A: Prompted reviewer (PDF → ICML review-form output).**
A wrapper. Works in an hour, wins nothing.

**T2-B: Agentic reviewer with verification tools (recommended for this track).**
The loop builds a reviewer that doesn't just opine — it *checks*:
- Parse paper → extract claims → verify: re-derive key math, run/inspect linked code, check citations exist via arXiv/Semantic Scholar API, check figures against claimed numbers
- Multi-persona reviewers (harsh theorist, empiricist, reproducibility cop) + AC meta-review synthesis
- Output in exact ICML review format (summary / claims&evidence / strengths / weaknesses / questions / scores / confidence)
- **Self-evaluation as backpressure:** the loop continuously scores itself against ground truth — decision accuracy + score correlation on held-out ICLR papers, error-detection rate on FLAWS-style injected errors — and iterates on its own review strategy to climb the metric. The demo is a *number going up on a W&B chart*, which is the single most persuasive autonomous-agent demo that exists.

**T2-C: Calibration angle.** Sub-option of B: explicitly target the "LLMs are too nice" problem — measure and fix positivity bias vs. human score distributions. Crisp narrative, cheap to add.

### Why Track 2 is structurally easier in 6 hours
No GPU, no flaky training runs, pure API + PDF parsing; ground-truth data exists; progress is a scalar metric the loop can hill-climb — the perfect Ralph backpressure signal. Track 1's bottleneck (real experiments take wall-clock time) eats loop iterations.

## 4. Recommended stack (either track)

- **Loop:** `ralph.sh` wrapping `codex exec --sandbox workspace-write --ask-for-approval never` (sponsor tool; or `claude -p` — decide by whose credits/limits we have). Iteration cap + stall watchdog + auto `git commit` per iteration.
- **Observability:** W&B Weave tracing on every iteration + W&B runs/tables for eval metrics. Sponsor is judging; a live dashboard is also our only window into the loop once hands-off.
- **Compute:** VESSL GPU only if Track 1 with small-model experiments; otherwise skip.
- **Repo skeleton (prepare TONIGHT, before the event):** `PROMPT.md`, `specs/` (detailed product spec — this is where we encode everything above), `fix_plan.md` seed, `AGENT.md`, eval data downloaded (ICLR reviews / FLAWS papers or experiment templates), API keys, `ralph.sh` tested for 2–3 iterations on a toy task.
- **Open rules questions to ask organizers at check-in:** how much may be pre-built vs. built by the loop? Which agent CLIs allowed? Is internet/API access during the loop unrestricted? Team size/merging?

## 5. Bottom line

1. **Highest win probability: Track 2, option T2-B** — verification-tool reviewer that hill-climbs a real metric (FLAWS error detection + ICLR decision accuracy) with the number-goes-up W&B demo.
2. **Highest ceiling / grand-prize story: Track 1, option T1-C** — micro-paper factory with a reviewer agent as backpressure; riskier, but "the loop produced 3 reviewed, compiled papers with real experiments" is a spectacular demo if it lands.
3. Either way, the *harness* is the product: spec quality, backpressure design, observability, and zero-touch discipline are what past winners were rewarded for.
